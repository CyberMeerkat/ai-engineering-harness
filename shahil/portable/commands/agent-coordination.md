# Agent Coordination

> Rules for how agents (subagents, background tasks, parallel executions) work together without creating drift, stale references, or silent failures. These rules are learned from production incidents — not theoretical.

## When This Skill Applies

This skill's rules apply whenever you:
- Launch multiple agents in parallel
- Use background agents (run_in_background)
- Delegate research to subagents and act on their findings
- Run batch operations that touch multiple state files
- Work in plan mode with agents

## Rules

1. **Parallel agents create stale cross-references.** When multiple agents run simultaneously and write to state files, their cross-references to each other's work may point to pre-completion state. After parallel agent execution, always reconcile state files — read each agent's output and verify cross-references are accurate. *(From: feedback_learned_parallel_agent_stale_refs)*

2. **One agent per repo + per concern.** The proven batch pattern is: launch one agent per repository AND per concern (e.g., "api security fixes" and "mobile brand fixes" are separate agents). 5 agents completed 3 options in ~90 seconds. Don't overload a single agent with cross-repo work. *(From: feedback_learned_multi_agent_batch_pattern)*

3. **Background agents get blocked by permissions.** Agents launched with `run_in_background` get blocked on Edit/Bash permission prompts with no way to respond. Use background agents for **research only** (Read, Grep, Glob). Apply edits in the parent context after the agent returns findings. *(From: feedback_learned_agent_permission_pattern)*

4. **Plan mode blocks agent writes.** Agents spawned during plan mode cannot use Write/Edit tools — plan mode restrictions propagate to child agents. Launch implementation agents only after ExitPlanMode. Use plan-mode agents exclusively for research and design. *(From: feedback_learned_plan_mode_blocks_agents)*

5. **3-pass cross-audit pattern.** For cross-cutting audits that touch multiple state files: Pass 1 — launch parallel research agents (read-only). Pass 2 — parent applies edits based on findings. Pass 3 — launch validation agent to confirm 0 discrepancies between state files. Never let agents edit state files in parallel — edits must be serialized through the parent. *(From: feedback_learned_cross_audit_pattern)*

6. **Never leave >10 uncommitted files overnight.** After batch operations that modify files across repos, commit per concern immediately. 45+ uncommitted files in mobile-app and 90+ in infra were at risk of loss. Group commits by concern (brand, security, docs), not by time. *(From: feedback_learned_uncommitted_cross_concern)*

7. **Multi-agent doc writing: Agent 2 must READ before appending.** When Agent 1 writes a file and Agent 2 must append to it, Agent 2 must READ the file first — then use Edit (not Write, which overwrites). The Write tool replaces the entire file. Pattern: Agent 1 writes → parent verifies → Agent 2 reads then appends via Edit. *(From: feedback_learned_multi_agent_doc_writing.md)*

8. **Plan agents are architects, not builders.** Plan agents lack Write/Edit tools. Use them to design structure and strategy, then pass their output as detailed implementation prompts to general-purpose agents. Never expect a Plan agent to create files. *(From: feedback_learned_plan_agent_scope.md)*

9. **Brief Plan agents with product/business context, not just code.** When launching Plan agents for product, marketing, or funnel tasks, code context alone is insufficient — code structure ≠ product structure. Before planning, run `/grill-me` to interview the user and capture the necessary business context (what each product IS, who it's for, how it makes money). Pass this context into Plan agent prompts alongside technical findings. *(From: feedback_learned_plan_agents_need_business_context.md)*

10. **Check working tree before spawning fix agents.** Before launching a worktree agent to fix a known risk or bug (R-xxx), run `git diff --stat` to check if fixes already exist as uncommitted changes. If changes exist in the relevant files, review and commit them directly instead of spawning an agent to re-derive the same fix. *(From: feedback_learned_check_uncommitted_before_fix_agent.md)*

11. **Specify exact monorepo paths in worktree agent prompts.** When spawning worktree agents that create new directories in a monorepo, explicitly state the exact target path (e.g., "create files at `apps/storefront/`, NOT at the repo root `storefront/`"). Agents default to repo root. Include the workspace layout in the prompt. *(From: feedback_learned_worktree_agent_paths.md)*

12. **Background agents can't write to `~/.claude/` paths.** The sandbox restricts agents to the project directory. Write/Bash to `~/.claude/commands/`, `~/.claude/agent_docs/`, or `~/.claude/hooks/` will be denied. Do global file writes in the main session, or pre-grant the path via `additionalDirectories` in settings.json before spawning. *(From: feedback_learned_agents_cant_write_global_paths.md)*

13. **Don't merge skills at different abstraction levels.** `/product-owner` (WHAT) → `/engineering-plan` (HOW) → `/director` (EXECUTE) is a responsibility chain, not duplication. PO owns sprint scope, EP creates technical plans for single features, Director dispatches cross-domain skill coordination. Merging them bloats one command with responsibilities at different scopes. *(From: feedback_learned_po_ep_director_chain.md)*
14. **Agent worktrees bypass when tools use absolute paths.** Write/Edit tools use absolute paths (e.g. `<projects>/{{project}}/infra/apps/...`) which bypass the worktree at `.claude/worktrees/agent-xxx/`. Do NOT use `isolation: "worktree"` for implementation agents that write code. Only use worktrees for read-only exploration agents. *(From: feedback_learned_worktree_absolute_paths.md)*
15. **Verify audit/research agent findings before acting.** Agent findings are hypotheses, not facts. Always read file:line before fixing what an agent reported. Agents may misidentify issues due to stale context or pattern-matching errors. *(From: feedback_learned_verify_audit_agent_findings.md)*
16. **Fall back to direct tools on context-mode NODE_MODULE_VERSION error.** If `ctx_batch_execute` fails with "NODE_MODULE_VERSION 115 vs 127", switch to Bash/Read/Grep immediately. Don't retry. Fix between sessions: `npm rebuild better-sqlite3` in the plugin directory. *(From: feedback_learned_context_mode_node_mismatch.md)*
17. **Never use `status` as a variable name in Monitor scripts — it is a zsh read-only builtin.** Assigning `status=$(...)` exits the Monitor script immediately with `(eval):1: read-only variable: status`. Use descriptive names: `run_state`, `ci_out`, `job_result`. Other zsh read-only builtins to avoid: `RANDOM`, `SECONDS`, `LINENO`. *(From: feedback_learned_monitor_zsh_status_var.md)*
18. **`sleep N && command` is blocked in the Bash tool — use Monitor with an until-loop or `run_in_background: true`.** The sandbox rejects commands starting with a long sleep. For "wait then check": Monitor with `until <check>; do sleep 20; done` fires a notification when the condition is met. For "run in background and be notified when done": Bash with `run_in_background: true`. For "resume after a delay": ScheduleWakeup. *(From: feedback_learned_sleep_blocked_use_monitor.md)*
19. **CLI version drift breaks built-in sub-agents — fall back inline, do not retry.** When a built-in sub-agent fails to load with `ENOENT` on a path containing a CLI version that does not match the running CLI (e.g. `~/Library/Caches/copilot/pkg/<arch>/<old-version>/definitions/<agent>.agent.yaml`), perform the work inline and tell the user the agent is unavailable due to version drift. Retrying re-fails identically. *(From: feedback_learned_cli_version_drift_subagents.md)*
20. **bash/grep/glob native-module ENOENT — abandon shell tools for the session, use file-API tools.** `pty.node` / `prebuilds/<arch>/rg` / other native-binary ENOENT inside the CLI cache means the binding is missing, not transient. Switch to `view`/Read instead of `cat`/`head`/`tail`, `edit`/`create`/Write instead of `sed`/`>` redirects, in-process Glob for filename patterns. Same family as ctx NODE_MODULE_VERSION (rule 16) — CLI cache drift. *(From: feedback_learned_native_module_enoent_fallback.md)*
21. **Rebind tool capabilities after mid-session toolset swap.** When a session notification, summary, model handover, or context compaction signals "tools changed", pause and explicitly map the prior toolset to the new one before continuing. Old tool-names called blindly waste a round-trip and risk silent fallbacks (e.g. `apply_patch`/`rg` → `edit`/`create`/`grep --output_mode`). *(From: feedback_learned_tool_swap_rebind.md)*
22. **`cbm-code-discovery-gate` blocks first Read on .md/.json/.yaml/.toml — batch with Glob/Grep or retry once.** Hook misclassifies plain text/state/config files as code and demands codebase-memory-mcp first. Workaround: include a Glob or Grep call in the same message, or retry the Read once — gate clears. Do not waste a turn calling codebase-memory-mcp on text/state files. *(From: feedback_learned_cbm_gate_first_read.md)*
23. **Detached launcher exit only proves the spawn command returned — verify the service itself separately.** When starting a long-lived process with `nohup`, detached Bash, or similar, the launching shell may report completion immediately even while the service remains healthy and serving traffic. After any detached launch, confirm readiness via the target port, health endpoint, or logs before diagnosing an outage or assuming startup succeeded. *(From: feedback_learned_detached_launcher_exit_is_not_service_exit.md)*
24. **In plan mode, never call `ExitPlanMode` in the same turn as posing an open question.** ExitPlanMode signals plan-final + ready for approval. Posting "Open questions before exit" / "Confirm" / "Should I" / "Decision needed" in the plan body THEN exiting forces user to either answer post-exit (no longer in plan-mode iteration) or reject the exit. Pre-flight check before every ExitPlanMode call: scan the plan body for unresolved-question patterns; if found, end the turn with questions visible and wait for user reply. Split "open questions" turns from "approval" turns. *(From: feedback_learned_plan_mode_no_auto_exit_after_question.md)*
25. **Pin CWD explicitly for `run_in_background=true` Bash calls — persistent shell CWD is fragile.** The Bash tool's persistent shell preserves CWD across calls but is not durable across model context shifts and can leak from unrelated prior tool calls (e.g. a Read on a different file). For long-running bg processes (servers, workers): always `cd /abs/path/to/proj && <cmd>` AND prefer absolute interpreter paths (`/abs/path/.venv/bin/python` not `python`). One-shot synchronous commands are less load-bearing because their failure surfaces immediately. Bad: `PYTHONPATH=src .venv/bin/python -m mymod`. Good: `cd /abs/path/to/proj && PYTHONPATH=src .venv/bin/python -m mymod`. *(From: feedback_learned_persistent_shell_cwd_unreliable_for_bg.md)*

26. **Never lead a clarification with "don't conflate" or similar correction phrasing.** When a user asks a clarifying question (especially around architecture choices, OAuth flows, naming, or scope), lead with the answer they expect — options, trade-offs, or a direct response. Phrases that imply user error ("don't conflate", "actually two things", "you're confusing X with Y", "to be clear, those are different") are out unless explicitly invited. Caveman/compressed-output modes amplify the snark — compression makes any tone-loaded phrase land twice as hard. Tone discipline is heightened, not relaxed, in those modes. *(From: feedback_learned_clarification_tone.md)*

27. **Compound shell loops (`do/done` blocks) ship as bash script files, not inline zsh strings.** Any iteration construct with `until`, `while`, or `for ... do ... done` invoked via the Bash tool's default zsh shell can fail with `(eval):1: parse error near 'done'`. Workaround: write the script to `/tmp/<name>.sh`, `chmod +x`, then `bash /tmp/<name>.sh`. Single-line `for x in ...; cmd "$x"; done` may work but compound bodies fail predictably. Prefer script-file invocation unconditionally. *(From: feedback_learned_zsh_compound_loops.md)*

28. **`page.evaluate` JS containing `return` must be IIFE-wrapped.** Bare top-level `return` in an evaluate body throws `Page.evaluate: SyntaxError: Illegal return statement`. Wrap multi-statement bodies as `(()=>{ ...statements...; return result; })()` or reduce to a single-expression form (`document.title`, `document.querySelectorAll('x').length`). Same applies when calling a Playwright-validation controller's `POST /action` with `action: "evaluate"`. *(From: feedback_learned_page_evaluate_iife.md)*

29. **Playwright pseudo-classes (`:has-text`, `:visible`, `:nth-match`) are invalid inside `document.querySelector`.** Those exist only in Playwright's locator API; the browser DOM rejects them with `Failed to execute 'querySelector' on 'Document': '...' is not a valid selector`. Inside `evaluate`, replace `:has-text("X")` with `Array.from(document.querySelectorAll('button')).find(b => /X/.test(b.textContent))`. Reserve Playwright pseudo-classes for `page.locator(...)` / `page.click(...)` calls only — not for any code that runs inside `evaluate` / browser DOM context. *(From: feedback_learned_evaluate_native_selectors.md)*

30. **SQLAlchemy `.as_string()` is the cross-dialect JSON text accessor — never use `.astext` when tests run on SQLite.** `Column.properties[key].astext` is Postgres JSONB only. When conftest patches JSONB→JSON for SQLite test fixtures, the indexed value becomes a plain `BinaryExpression` whose comparator has no `astext` attribute → `AttributeError: Neither 'BinaryExpression' object nor 'Comparator' object has an attribute 'astext'`. `Column.properties[key].as_string()` emits PG `->>` and SQLite `json_extract(col, '$.key')`; equality and `like` work cross-dialect. Reach for `.as_string()` first on any JSON/JSONB filter or projection. *(From: feedback_learned_sqlalchemy_as_string_cross_dialect.md)*

31. **`select(...).distinct(col1, col2)` is Postgres `DISTINCT ON` only — emits broken DISTINCT on SQLite.** SQLite falls back to plain `DISTINCT` across all selected columns, returning every distinct row instead of one per group. For "latest row per (k1, k2)" portable across PG + SQLite, use a subquery-join: `latest = select(M.k1, M.k2, func.max(M.created_at).label('mx')).group_by(M.k1, M.k2).subquery(); stmt = select(M).join(latest, and_(M.k1 == latest.c.k1, M.k2 == latest.c.k2, M.created_at == latest.c.mx))`. Window function `row_number().over(...)` also works cross-dialect on PG 8+ / SQLite 3.25+ but the subquery-join is the simplest form. *(From: feedback_learned_distinct_on_portability.md)*

32. **`httpx` multi-value query params need list-of-tuples — dict collapses duplicate keys.** For endpoints accepting `?include=a&include=b` style (FastAPI `list[str] = Query(default=None)`, OpenAPI `array` query), Python dict cannot hold duplicate keys: `{"include": "a", "include": "b"}` collapses to one entry → only one value is sent → the endpoint silently returns a smaller projection. Build params as `[("include", "a"), ("include", "b"), ("pool", pool)]` and pass to `client.get(url, params=...)`. Tests should assert on tuples shape: `assert "enriched_fields" in [v for (k, v) in params if k == "include"]`. *(From: feedback_learned_httpx_multi_value_params.md)*

33. **Agent-framework tool-exception wrapping turns hard infra failures into silent "no data" patterns.** `BaseAgent`-style frameworks wrap tool exceptions as strings passed to the LLM (`[Tool error: gmail_create_draft raised EnvironmentError: ...]`). The LLM treats them as context and proceeds blind. Found in `delta-ai-funnel-accelerator/agents/base.py:276-280` — production Gmail was silently non-functional for an unknown period; no errors visible in logs. When adding any tool that talks to infrastructure (DB, HTTP, IMAP, OAuth, SMTP, file I/O), write a NON-LLM smoke test (`python -c "from tools.X import Y; Y(...)"` style) that fails loud at the boundary. Never rely on agent runs alone to validate infra connectivity. *(From: feedback_learned_agent_tool_exception_wrapping_hides_infra.md)*

34. **Secret pasted in chat = exposure event; rotate immediately post-validation.** When asking the user for a secret (OAuth client_secret, API key, password), pre-stage a `read -s -p "Paste: " SECRET` snippet IN THE PROMPT before they type. If the secret lands in chat anyway: acknowledge the exposure openly, use the exposed value to finish the immediate task, schedule rotation as the immediate next step. Never write the exposed secret value into a subsequent tool call (Bash heredoc, Write tool) — that's adding a second exposure surface for no benefit. Plain question — "what's the secret" — invites paste-into-chat; framed question + snippet protects. *(From: feedback_learned_secret_paste_in_chat_is_exposure_event.md)*

35. **CWD does not persist between Bash tool invocations — always prefix `cd /absolute/path && ...` or use absolute paths.** A `cd` in one Bash call does NOT carry to the next. Symptom: `open <projects>/X/docker-compose.yml: no such file or directory` after a previous `cd <projects>/X/sub && cmd` succeeded. Hard rule when working from a subdirectory across multiple Bash calls: include the `cd /abs/path &&` prefix in EVERY call, or use absolute paths inside the command (e.g. `docker compose -f <projects>/X/sub/docker-compose.yml ...`). Never assume cwd persists. *(From: feedback_learned_cwd_doesnt_survive_bash_calls.md)*

## Anti-Patterns

| Pattern | Problem | Fix |
|---------|---------|-----|
| Background agent does edits | Blocked by permissions, hangs | Research only in background |
| Parallel agents edit same state file | Last write wins, data lost | Serialize edits through parent |
| Trust subagent severity claims | Agents overstate severity | Read source before fixing |
| Trust subagent npm/registry claims | Agents hallucinate versions | Verify with `npm view` directly |
| Skip reconciliation after parallel run | Cross-references point to stale state | Always reconcile after |
| Batch edit without committing | 10+ files at risk overnight | Commit per concern immediately |
| Spawn fix agent without checking git diff | Agent re-derives existing fixes | Check uncommitted changes first |
| Omit monorepo path in agent prompt | Agent creates at repo root | Always specify apps/<name>/ path |
