## Learned Rules — Infrastructure

11. **Docker COPY overwrites generated Prisma client.** `.dockerignore` MUST exclude `node_modules`. If host `node_modules` are copied into the image, the generated Prisma client (with platform-specific binaries) is overwritten by the host version (wrong platform). *(From: feedback_learned_docker_copy_overwrites_generated)*
12. **npm prune after `prisma generate` corrupts the client.** Never run `npm prune --production` after `prisma generate` — it removes `@prisma/engines` which were just generated. *(From: feedback_learned_npm_prune_breaks_prisma)*
13. **API startup delay is 15-20 seconds.** After container restart, wait or poll before testing. `curl` will get connection refused during startup. *(From: feedback_learned_api_startup_delay)*
14. **VPS api/ needs .dockerignore on disk.** Stale `node_modules` on VPS disk get copied into Docker context. Verify `.dockerignore` exists at the repo root on VPS after initial clone. *(From: project_learned_vps_dockerignore)*
15. **Stash VPS changes before git pull.** VPS accumulates hotfix changes from manual edits. Always `git stash` before `git pull` to avoid merge conflicts. Verify after: `git stash list`, drop if safe. *(From: feedback_learned_vps_stash_before_pull)*
16. **Prune git refs before VPS pull after branch renames.** Stale remote refs block pull. Run `git remote prune origin` first. *(From: feedback_learned_git_ref_prune)*
17. **Run `prisma migrate resolve --applied` after db:push drift.** When migrations exist but schema was managed via `db push`, migrations show as "not applied". Run `prisma migrate resolve --applied <name>` before `migrate deploy`. *(From: feedback_learned_prisma_migrate_resolve)*
18. **vue-gen has no automated deploy pipeline.** No deploy key, no CI, no webhook. Needs EP to set up proper git-based deployment (deploy key on VPS, or CI image push). Current workaround (docker cp) violates safety rules. *(From: project_learned_vuegen_no_deploy_pipeline)*
19. **VPS deploy keys are per-repo.** Each nested repo (api, mobile-app, vue-gen) needs its own SSH deploy key with a separate `Host` alias in `~/.ssh/config`. The api key is `github-api` → `~/.ssh/id_pod_api`. Don't assume one key works for all repos. *(From: feedback_learned_vps_deploy_key_scope)*
20. **Deploy webhook is per nested repo.** GitHub webhooks fire per-repo, not per-directory. The api webhook (ID 604347394) only triggers on api pushes. Mobile-app and vue-gen need their own webhooks or a unified deploy strategy. *(From: project_learned_webhook_per_nested_repo)*
21. **Webhook --agents flag runs 26 containers despite EP-008.** The deploy.sh `--agents` flag was not removed from the webhook config after intelligence was disabled. Verify webhook command doesn't include flags for disabled services. *(From: project_learned_webhook_agents_flag)*
22. **VPS health endpoint at localhost, not nginx.** `/api/v1/health` returns 404 via nginx proxy. Use `curl http://localhost:3001/api/v1/health` from inside VPS, or use `scripts/smoke-test.sh` which tests internally. *(From: project_learned_health_endpoint_path, project_learned_health_route_not_proxied)*
23. **VPS needs explicit seed after migrate.** Migrations create tables but don't populate data. After `prisma migrate deploy`, run the seed script separately: `node prisma/seed.js` (not `npx prisma db seed` which swallows stdout). *(From: project_learned_vps_seed_after_migrate, feedback_learned_prisma_seed_stdout)*
24. **Verify VPS state via SSH, not state files.** Triage deploy sections ("N commits undeployed") go stale instantly. Before any deploy decision, compare `git log --oneline -5` on VPS vs local to see actual drift. *(From: feedback_learned_deploy_state_stale, feedback_learned_deploy_commit_count)*
25. **VPS >5 days behind = HIGH risk.** Verify deploy currency at every sprint acceptance. If VPS is >5 days behind HEAD, flag as HIGH risk in dev-manager risk register. *(From: feedback_learned_deploy_gap_threshold)*
26. **Never SCP files to a git-managed VPS.** Production receives code via `git pull` only. SCP creates uncommitted local changes that block `git pull --ff-only` with `error: Your local changes would be overwritten by merge`. If bootstrapping a self-referential change, SCP only the minimum, then commit+push+pull for everything else. *(From: 2026-04-06 — SCP'd docker-compose.prod.yml caused deploy failure)*
27. **`docker compose restart` does not reload env vars.** Use `docker compose up -d <service>` to recreate when env vars change. `restart` reuses the same container with old env — new vars are silently missing. *(From: 2026-04-06 — SLACK_WEBHOOK_DEPLOYS not picked up after restart)*
28. **Pin Prisma CLI version to match package.json.** Never use bare `npx prisma` — a globally cached v7 will break v6 schema with "The datasource property `url` is no longer supported." Use `npx prisma@$(node -p "require('./package.json').devDependencies.prisma") generate`. *(From: feedback_learned_npx_prisma_version_pinning)*

29. **Host bind mounts override Dockerfile-created directories.** A `volumes:` bind mount makes the Dockerfile's `mkdir` invisible — the host directory (possibly empty) replaces the container directory. Create required directories on the host, not just in the Dockerfile. *(From: feedback_learned_bind_mount_overrides_dockerfile.md)*
30. **Test webhooks from inside the container.** Signature-verified webhooks (PayFast ITN, Stripe) fail when sent through nginx/proxies that re-encode the body. Use `docker exec <container> wget http://127.0.0.1:<port>/...` to preserve exact body encoding. *(From: feedback_learned_container_webhook_testing)*

31. **Docker Compose `volumes: []` does NOT clear base volumes.** Compose merges sequence fields by appending — `volumes: []` adds nothing but doesn't remove. Never put dev bind mounts in the base compose. Use `docker-compose.override.yml` for dev volumes (auto-loaded in dev, skipped when prod uses explicit `-f` flags). *(From: feedback_learned_compose_volumes_override.md)*

32. **Next.js standalone Dockerfile needs `mkdir -p public`.** Empty `public/` directories aren't tracked by git. The `COPY --from=builder /app/public ./public` step fails with "not found". Always add `RUN mkdir -p public` before the build step. *(From: feedback_learned_nextjs_standalone_public_dir.md)*

33. **Compose `environment:` overrides Dockerfile `ENV`.** If base compose sets `NODE_ENV: development`, the prod override MUST explicitly set `NODE_ENV: production` — the Dockerfile's `ENV NODE_ENV=production` is silently ignored. *(From: feedback_learned_compose_env_overrides_dockerfile.md)*
34. **Socket.IO through ngrok/tunnels needs polling-first transport.** Set `transports: ["polling", "websocket"]` — WebSocket-first fails silently through ngrok free tier. Players connect via HTTP but socket never establishes. *(From: feedback_learned_ngrok_socketio_polling_first)*
35. **Next.js 16+ blocks dev HMR from non-localhost origins.** Add tunnel domains to `allowedDevOrigins` in `next.config.ts` (e.g. `["*.ngrok-free.app"]`). Error: `Blocked cross-origin request to /_next/webpack-hmr`. *(From: feedback_learned_nextjs_allowed_dev_origins)*
36. **Workspace packages in Docker need symlinked node_modules.** When building a Docker image for an app consuming npm workspace packages: (1) merge workspace package deps into consumer's package.json, (2) add bundler aliases to source dirs, (3) `ln -sf /build/apps/<consumer>/node_modules /build/packages/<each-pkg>/node_modules`. Without symlinks, workspace source files can't resolve transitive deps (`tailwindcss`, `@prisma/internals`). *(From: feedback_learned_workspace_docker_symlinks.md)*
37. **Never use `|| true` on Docker RUN steps that produce required output.** A failed command masked by `|| true` caches as success. Subsequent `--no-cache` may still serve cached empty output from BuildKit. Use `docker builder prune -af` to truly bust cache. Only use `|| true` for genuinely optional steps. *(From: feedback_learned_docker_no_cache_or_true.md)*
38. **Run `prisma generate` as root after db:push in Docker containers.** Prisma generate fails with EACCES when run as non-root inside containers because it writes to `node_modules/.prisma/`. Either run as root or rebuild the container after schema changes. *(From: feedback_learned_prisma_docker_eacces.md)*
39. **Docker build may cache stale Prisma client after schema changes.** After adding new models, `docker compose build` may cache the old Prisma client. Verify new models work after rebuild — if not, use `--no-cache` or run `prisma generate` inside the running container. *(From: feedback_learned_docker_prisma_client_cache.md)*
40. **Splitting Docker services requires distinct named volumes.** When splitting one service into two, never share a `node_modules` volume — different `package.json` files will cause dependency conflicts. Create separate named volumes (e.g. `erp_node_modules`, `admin_node_modules`). *(From: feedback_learned_docker_split_volumes.md)*
41. **`gh pr merge` may not trigger deploy webhooks — always verify VPS state.** GitHub API-driven merges may not generate the same push event that triggers webhooks. After any merge to main, check VPS: `ssh root@<VPS> "cd /opt/<project> && git log -1 --oneline"`. If stale, deploy manually with `git pull origin main && docker compose up -d`. *(From: feedback_learned_verify_webhook_after_api_merge.md)*

42. **`docker compose up --force-recreate` reuses the existing image tag — pair with `--build` when swapping Dockerfiles.** Force-recreate recreates the CONTAINER but doesn't rebuild the image. If you're switching between compose files that reference different Dockerfiles (dev vs prod), pass `--build` or the container will boot from whatever image the previous successful build produced. Symptom: container crashes in a way that matches the OTHER compose's command. Diagnose with `docker inspect <container> --format '{{.Config.Image}}'`. *(From: feedback_learned_compose_recreate_needs_build.md)*

43. **GitHub webhook `status_code=0` means the receiver is unreachable — check deliveries before container logs.** When a webhook silently stops firing, first run `gh api repos/<owner>/<repo>/hooks/<id>/deliveries --jq '.[0:5] | .[] | "\(.delivered_at) status=\(.status)(\(.status_code))"'`. `status_code=0` = HTTP call never got a response (nginx/proxy down, DNS broken, port conflict). The webhook container's own logs will be silent because no request arrived. After fixing reachability, redeliver: `gh api --method POST .../hooks/<id>/deliveries/<delivery_id>/attempts`. *(From: feedback_learned_github_webhook_status_zero.md)*

44. **nginx resolves upstream hostnames at startup — transient DNS lag causes `[emerg] host not found in upstream`.** Even when `depends_on: service_healthy` passes, Docker's internal DNS can lag a few seconds behind container startup. nginx caches upstream resolution once at boot and dies immediately on miss. Quick fix: retry `docker compose up -d --force-recreate nginx`. Permanent fix: variable-based `proxy_pass` with `resolver 127.0.0.11 valid=10s` so nginx re-resolves per request. *(From: feedback_learned_nginx_upstream_dns_startup.md)*

45. **For remote builds/deploys >2 min, run polling loops server-side — don't pipe `tail -f` through SSH.** SSH streams drop before 4.5-min Docker builds complete, failing the Monitor task. Use `ssh host "until grep -qE 'Deploy complete|Deploy FAILED' \$LOG; do sleep 15; done; tail -30 \$LOG"` — the until-loop runs on the remote host, so the SSH connection just needs to stay alive for the exit. *(From: feedback_learned_ssh_monitor_unreliable.md)*

46. **bitnami/* Docker Hub images are deprecated and removed (sunset 2025).** Pulls fail with `manifest unknown`. For CI/dev/prod default to official upstream: MinIO → `quay.io/minio/minio:RELEASE.YYYY-MM-DDTHHMMSSZ`, Postgres → `postgres:17-alpine` or `pgvector/pgvector:pg17`, Redis → `redis:7.2-alpine`. If you need bitnami-style features (auto bucket creation), implement them as a separate setup step (e.g. `mc mb` for MinIO buckets). *(From: feedback_learned_bitnami_images_dead.md)*

47. **GitHub Actions service containers cannot pass CMD args.** The `services:` block has no `command:` or `args:` field — only `image`, `env`, `ports`, `volumes`, `options`. For images that need a subcommand to start (`minio server /data`, custom entrypoints), use a `docker run -d` step BEFORE tests instead. Service containers only work for images that boot correctly with their default CMD (postgres, redis, mysql, etc.). *(From: feedback_learned_gh_actions_no_cmd_args.md)*

48. **Never `prisma db push` against shared environments — it silently accumulates schema drift.** `db push` mutates the schema without creating a migration file. The dev/staging/prod DB stays in sync but CI fresh-DB deploys diverge silently. Surfaces only when a fresh-DB integration test queries a column that exists in `schema.prisma` but no migration created. Always use `npx prisma migrate dev --name <descriptive>` so a tracked migration is committed. Quarterly drift audit: `prisma migrate diff` against a shadow Postgres. *(From: feedback_learned_db_push_drift.md)*

49. **Drift consolidation workflow needs shadow DB + DROP filtering + idempotent wrapping.** `prisma migrate diff` requires `--shadow-database-url` (use disposable `docker run -d --rm` Postgres). Strip every `DROP TABLE/TYPE/CONSTRAINT/FOREIGN KEY` line from the output (review separately as data-aware migrations). Make additive ops idempotent: `CREATE TABLE/INDEX IF NOT EXISTS`, `ADD COLUMN IF NOT EXISTS`, wrap `CREATE TYPE` and `ALTER TYPE ADD VALUE` and `ADD CONSTRAINT FK` in `DO $$ BEGIN ...; EXCEPTION WHEN duplicate_object THEN NULL; END $$;`. Verify by applying twice — second pass must produce zero errors. *(From: feedback_learned_prisma_migrate_diff_workflow.md)*

50. **In GitHub Actions, `$GITHUB_ENV` writes only propagate to *subsequent* steps — use `export` for same-step subprocess access.** Writing `echo "VAR=$val" >> "$GITHUB_ENV"` inside a step does NOT make VAR available to python3/bash subprocesses called later in the same step. Those raises KeyError / empty. Both `export VAR` (current-step scope) AND `$GITHUB_ENV` write (downstream-step scope) are required when a var is needed in both contexts. *(From: feedback_learned_github_env_same_step.md)*

51. **macOS-13 GitHub Actions runners queue 6+ hours — never add darwin-x64 to a CI matrix without a confirmed Intel Mac user.** darwin-arm64 (macos-14) finishes in under 2 min; darwin-x64 (macos-13) stays "queued" indefinitely and blocks matrix completion signals. Remove darwin-x64 from the matrix and expose it as a `workflow_dispatch` input for on-demand builds. If it's already queued and blocking, cancel the run. *(From: feedback_learned_macos13_runner_queue.md)*

52. **Daemon "successfully started" log does NOT mean port is bound — verify with `lsof -aPi -p $PID | grep LISTEN`.** Process can be alive (`ps -ef` finds it) yet bind to a different port (env mismatch) or no port (config error). Wasted minutes on `curl :3000` returning Connection refused after seeing the success log; `lsof -aPi -p 95490` revealed actual bind on `:3100`. After starting any backend daemon, enumerate its bound ports via `lsof` and match against expected from `.env` — never against hardcoded assumptions in CLAUDE.md or prose docs. *(From: feedback_learned_daemon_started_vs_port_bound.md)*

53. **Open the SaaS dashboard headful before guessing — 2+ failed CLI/GraphQL guesses = stop and look.** When a managed PaaS (Railway/Vercel/Heroku/Render/Netlify) deploy is broken and 2 remediation attempts via CLI/GraphQL haven't fixed it, launch headful Playwright/browser, navigate Service → Settings → Source/Deploy History, and read the actual displayed values. GraphQL queries can hide or omit fields the UI shows (e.g. tracked branch). One headful glance fixed a 6-iteration "alembic boot loop" debug in 30 seconds — the worker was wired to the wrong git branch. *(From: feedback_learned_playwright_headful_for_paas_dashboards.md)*

54. **Step 1 of any "stale code" deploy debug = confirm `(repo, branch, deployed commitSha)`.** Before reading any deploy log line, verify the service is actually wired to the branch you've been pushing. Railway: GraphQL `serviceInstance { source { repo image } }` shows repo, but **branch** must be confirmed via dashboard Settings or `serviceConnect` payload. Vercel: project Settings → Git. If the deployed commit ≠ `git rev-parse HEAD` of your branch, fix wiring first — don't read another log line. Lost an hour on an "alembic missing revision" error that was simply the worker tracking a stale `main` branch. *(From: feedback_learned_check_git_source_before_deploy_debug.md)*

55. **`railway redeploy` reuses the OLD deployment snapshot — UI overrides set after that snapshot will NOT take effect.** Redeploy reruns the same baked-in startCommand, dockerfile path, and env values. To apply new config: push a commit to the tracked branch OR call GraphQL `serviceInstanceDeployV2(serviceId, environmentId, commitSha)`. After any redeploy, sanity-check `meta.startCommand` on the new deployment matches expected. Generalises: any "redeploy/restart" command on a PaaS may not be a fresh build — confirm before assuming config picked up. *(From: feedback_learned_railway_redeploy_reuses_snapshot.md)*

56. **Before editing any root deploy config in a multi-service monorepo, enumerate every consumer.** `railway.{toml,json}`, `Procfile`, `nixpacks.toml`, `Dockerfile`, `vercel.json`, root `package.json` scripts are shared contracts. Before editing, list every PaaS service whose source repo matches (Railway: GraphQL `services { source { repo } }`). If >1 service consumes the file, use **per-service overrides** (`serviceInstanceUpdate` for Railway, project settings for Vercel) instead of editing the shared file. Edited a shared `railway.toml` to fix one worker — would have broken 4 other services on next deploy if not reverted in 2 minutes. *(From: feedback_learned_enumerate_shared_config_consumers.md)*

57. **Use Railway GraphQL `deploymentLogs` in scripts — `railway logs` needs an interactive TTY.** `railway logs --deployment <id>` returns nothing in non-interactive contexts because it's a streaming command. For automation, POST to `https://backboard.railway.com/graphql/v2` (token at `~/.railway/config.json` `.user.accessToken`) with `deploymentLogs(deploymentId, limit)`. When `deploymentLogs` truncates (~11 lines), fall back to `environmentLogs(environmentId, filter, beforeLimit)` for broader runtime traces. *(From: feedback_learned_railway_logs_use_graphql_in_scripts.md)*
58. **When Railway dashboard metadata is flaky, prove deploy state from public behavior first.** React-heavy PaaS dashboards can hide or delay commit/branch metadata (`networkidle` never settles, `domcontentloaded` returns before useful state renders). Before trusting the UI, probe `/healthz`, `/readyz`, route presence, and the exact changed behavior on the public service. Use dashboard deploy history as supporting evidence only. *(From: feedback_learned_validate_railway_from_public_behavior.md)*

59. **Alembic multi-head merge: generate the merge file via container with the host versions/ volume-mounted, then rebuild the image.** When a containerized migrate service fails with `Multiple head revisions are present for given argument 'head'`, do NOT swap `head` for `heads` (advances all leaves independently, makes future migrations diverge further). Instead: (1) `docker compose run --rm <migrate-svc> bash -lc 'alembic heads'` to confirm head count, (2) `docker compose run --rm -v "$(pwd)/src/<pkg>/alembic/versions:/app/src/<pkg>/alembic/versions" <migrate-svc> bash -lc 'alembic merge -m "<msg>" <head1> <head2>'` to generate the merge file on the host (volume mount required — without it the file is lost on container exit), (3) `docker compose down && docker compose up -d --build` so the new file is baked into the long-lived migrate service's image. Inspect revision/down_revision DAG first to confirm merge is the right call vs picking one head and reverting the other. *(From: feedback_learned_alembic_multihead_merge_via_container.md)*

60. **`docker-compose` reads `.env` keys ONLY when the compose file substitutes them via `${VAR}` — per-service `environment:` blocks override.** `.env` is NOT a wholesale env source for compose-managed services; only keys referenced via `${X}` or `${X:-default}` are loaded. Hardcoded per-service `environment:` values override `.env`. Stale or wrong values in `.env` for unreferenced keys are harmless at runtime but mislead audits. Before auditing a compose project's `.env`: `grep -nE '\$\{' docker-compose.yml` to identify the actual subset compose reads. Anything in `.env` NOT in that grep is dead config relative to the compose stack. **Caveat:** if the same service is also runnable via `python -m`/`uvicorn` directly (without compose), the full `.env` IS read by the app's settings loader — audit both pathways separately. *(From: feedback_learned_docker_compose_env_substitution.md)*

61. **`npm install` symlinks local `file:` deps; their transitive deps don't resolve without `--install-links`.** A symlinked `file:` package resolves its own `dependencies` from ITS directory tree, not the consumer's `node_modules` — so transitive deps go missing. Symptom: `Module not found: Can't resolve '<X>'` where X is a dependency of the `file:` package, not of your app. Fix: `npm install --install-links` copies `file:` deps in as real folders so node's upward resolution reaches the consumer's `node_modules`. Applies whenever you install (or compose-command-install) a local `file:` package with a non-empty `dependencies` block. *(From: feedback_learned_npm_install_links.md)*

62. **A Next.js (or other SSR) app in Docker needs TWO API base URLs.** Server components / SSR reach sibling containers by Docker-network hostname (`http://<service>:<port>`); browser code uses the host-published URL (`http://localhost:<port>`). A single `localhost`-based URL works in the browser but breaks every server-side fetch — symptom: SSR pages 500 with `ECONNREFUSED` (often after a long hang) while browser-only paths work. Fix: an internal/server env var and a public/browser env var, selected on `typeof window === 'undefined'`. *(From: feedback_learned_nextjs_ssr_docker_url.md)*

63. **Dry-run DB-mutating operations on a `pg_dump` scratch copy before touching a populated DB.** Before `prisma migrate deploy`, a data backfill, or a destructive schema change against a DB with real data: `pg_dump` the real DB, restore into a throwaway database, apply the change there, and proceed against the real DB only once the scratch run is clean. Surfaces non-idempotent migrations, colliding backfill values, and latent integrity violations (e.g. duplicate-key rows a nominally-valid unique index wasn't actually enforcing — exposed when the restore rebuilds the index) at zero risk. The dump doubles as the backup. *(From: feedback_learned_pgdump_scratch_dryrun.md)*

64. **`npm install` cannot resolve the `workspace:` protocol — rewrite to `file:` first in Docker builds.** `workspace:*` is pnpm-only; `npm install` hard-errors `npm error code EUNSUPPORTEDPROTOCOL` / `Unsupported URL Type "workspace:"`. Any npm-based Docker build of a package that lives in a pnpm workspace must rewrite or strip `workspace:*` deps before install. Next.js: vendor the package into the build context, rewrite `workspace:*`→`file:/<path>`, then `npm install --install-links` (see rule 61). Vite (aliases `@scope/*` to source): strip every `workspace:` dep from `package.json` before `npm install`. A stale "no workspace deps" Dockerfile comment is not trustworthy — grep `package.json` for `workspace:`. *(From: feedback_learned_npm_workspace_protocol_docker.md)*

65. **A `redis-cli` healthcheck without `-a` is false-healthy on a password-protected Redis.** `redis-cli` exits 0 even on error *replies* (`NOAUTH Authentication required`, `WRONGPASS`) — only non-zero on a connection failure. So a healthcheck `redis-cli --raw incr ping` (no `-a`) reports the container "healthy" regardless of whether auth works. Always authenticate the healthcheck: `redis-cli -a "${REDIS_PASSWORD}" --raw incr ping`. When a redis container shows `healthy` but a client reports an auth error, distrust the healthcheck — a "healthy" redis only proves the process is up, not that clients can connect. *(From: feedback_learned_redis_healthcheck_needs_auth.md)*

66. **`docker compose config` aborts on `${VAR:?}` required vars missing from the local `.env` — use `--no-interpolate`.** Validating a multi-file compose merge from a dev machine fails with `required variable <X> is missing a value` when the prod overlay references a `${VAR:?}` secret absent locally; `config` resolves interpolation for the whole file before emitting anything. `docker compose -f a.yml -f b.yml config --no-interpolate` merges + validates without resolving `${VAR}` — the merged image/command/volumes structure is still fully inspectable. To extract one service's block: `awk '/^  <svc>:/{f=1} f{print} f&&/^  [a-z]/&&!/<svc>/{exit}'`. *(From: feedback_learned_compose_config_no_interpolate.md)*

67. **CI green does not mean the prod Docker build passes — build prod images locally before a cutover.** CI typically runs unit/lint/typecheck/integration; it does NOT build the production `Dockerfile.prod` images, so those stay unverified. Before a prod cutover, build every prod image locally (`docker build -f apps/<app>/Dockerfile.prod .`). Iterate on Dockerfile fixes with local builds, never with prod deploy cycles — each prod deploy is slow and a mid-way failure can leave a half-deployed state. *(From: feedback_learned_ci_green_not_prod_docker.md)*

68. **Railway `*.railway.internal` hostname = the service's original create-time name; survives later rename — never compose the internal URL from the current service name.** When a Railway service is renamed in the dashboard, its private domain stays legacy. Setting `<consumer>.ORCH_WRITE_URL=http://tds-network-orch-write.railway.internal` against a service whose current name is `tds-network-orch-write` but whose `RAILWAY_PRIVATE_DOMAIN` is still `tds-enrich-match-orchestrate.railway.internal` fails with `httpcore.ConnectError: [Errno -2] Name or service not known`. Two follow-up landmines once the correct host resolves: (a) internal URLs default to TCP service port — if the app listens on `:8010` but the public-port mapping points at a different port, `Errno 111 Connection refused`. Always include explicit `:<port>` for internal URLs. (b) Public HTTPS URL pattern `https://<service>-production.up.railway.app` auto-routes to the right internal port and sidesteps both issues — slightly higher egress cost, but the proven pattern across this project's consumers (CRM-Manager → orch-write traffic uses public URL). Look-up command: `railway variables --service <name> --kv | grep RAILWAY_PRIVATE_DOMAIN`. *(From: feedback_learned_railway_internal_hostname_persists.md)*

69. **GCP OAuth 2.0 client secrets are write-once — cannot download after creation; plan workflows around "Add Secret" + capture immediately.** GCP Console (2025+) shows: `Viewing and downloading client secrets is no longer available. If you have lost the secret below, please add a new one.` The only path is "Add secret" on the client detail page, which generates a NEW secret shown once. Old secrets keep working until manually deleted (zero downtime). Two patterns: (a) Fresh setup: capture secret at OAuth client creation time (downloaded JSON includes it). (b) Lost secret: Add Secret → copy immediately → delete old. NEVER prompt a user to "look up" an existing OAuth secret in GCP — walk them through Add Secret + capture instead. *(From: feedback_learned_gcp_oauth_secrets_write_once.md)*

70. **`docker compose restart` does NOT re-read env_file — use `up -d --force-recreate <service>`.** Restart is `docker stop` + `docker start` on the existing container, retaining env vars set at container-create time. After modifying `.env` (or any file referenced by `env_file:`), `docker compose -f docker-compose.yml restart <svc>` will continue using stale env. The fix: `docker compose -f docker-compose.yml up -d --force-recreate <service>`. Verify env-var key presence post-recreate via `docker compose exec -T <svc> env | grep -Eo '^<KEY_PREFIX>_[A-Z_]+'` (names only — don't print secrets). *(From: feedback_learned_docker_compose_restart_doesnt_reread_env_file.md)*
